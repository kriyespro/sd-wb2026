# Winning Blueprints — Development Plan
# Source: durga.txt
# Goal: Build one platform with three isolated experiences sharing the same backend

================================================================================
BRAND POSITIONING (NON-NEGOTIABLE)
================================================================================

Primary identity: Professional Digital Growth Company (AI Growth Partner)
Secondary identity: Internal Industry Academy (talent pipeline — NOT the headline)

Client-facing message:
  "Grow Your Business with AI, Digital Marketing & Web Technology"

Student-facing message (Academy pages only):
  "Become an Industry Ready Digital Marketing Professional"

Model reference: TCS / Infosys / Zoho — company first, training internal.

================================================================================
PLATFORM ARCHITECTURE
================================================================================

Three isolated experiences, one Django backend:

1. PUBLIC AGENCY WEBSITE (primary brand)
   - Audience: Business owners
   - Goal: Leads → sales → projects
   - No student/training emphasis on homepage

2. ACADEMY PORTAL (secondary brand)
   - Audience: Students, career switchers
   - Goal: Admissions, learning, internship, placement
   - URL: /academy/ or academy.winningblueprints.in

3. INTERNAL OPERATIONS PORTAL
   - Audience: Staff, mentors, PMs, interns
   - Goal: Client delivery + student training pipeline
   - URL: /ops/ or internal subdomain (staff-only)

Shared backend:
  - Single user model with RBAC
  - Shared project/client data
  - Strict permission boundaries per portal

================================================================================
PHASE 4 — CLIENT DASHBOARD
================================================================================

Clients see professional delivery only. No student/intern info.

Modules:
  [ ] Projects (status, milestones)
  [ ] Reports (SEO, ads, analytics summaries)
  [ ] Invoices (link to billing phase)
  [ ] Meetings (scheduled calls)
  [ ] Files (deliverables — mentor-approved only)
  [ ] Support (tickets/messages to PM)
  [ ] Analytics (read-only metrics)
  [ ] Goals
  [ ] ROI summary

Rules:
  - Deliverables visible to client ONLY after mentor/QA approval
  - Students never appear in client UI
  - All client communication goes through PM / Account Manager

Models:
  - ClientAccount (linked to user)
  - Project, Milestone, Deliverable
  - Report, Meeting, SupportTicket
  - File (with approval_status field)

================================================================================
PHASE 5 — INTERNAL OPERATIONS PORTAL
================================================================================

Hidden from public. Staff-only talent pipeline + project delivery.

Agency delivery workflow:
  Lead → Sales → Business Consultant → Project Manager → Senior Expert →
  Student Team → Quality Check → Delivery

Talent pipeline module:
  [ ] Applications
  [ ] Assessments
  [ ] Training progress
  [ ] Mentor allocation
  [ ] Project allocation (student/intern to project)
  [ ] Performance tracking
  [ ] Graduation
  [ ] Hiring (intern → employee)

Project assignment rules (enforce in code):
  - Every project: 1 PM (owner), 1 Senior Specialist (tech lead)
  - Optional interns/students for execution
  - Students CANNOT message clients unless explicitly authorized
  - All deliverables require mentor approval before client visibility

Team structure (internal view):
  CEO → COO → Agency Director
  Teams: SEO, Google Ads, Meta Ads, Development, AI Automation, Content, Design
  Each team: Senior → Executive → Intern → Student

Models:
  - Team, TeamMember
  - ProjectAssignment (user, project, role: pm|senior|intern|student)
  - MentorAllocation
  - PerformanceReview
  - Application, Assessment

================================================================================
KEY BUSINESS RULES (ENFORCE IN CODE)
================================================================================

1. Homepage = agency only. No student language above fold.
2. Academy is nav item #9, not #1.
3. Client dashboard never exposes student/intern identities.
4. Deliverables require mentor approval before client sees them.
5. Students do not communicate with clients by default.
6. All data scoped to request.user (multi-tenant safety).
7. Revenue features (leads, projects, client portal) before academy depth.
